# Back-end 文档格式要求

## 1. 适用范围
本规范适用于 back-end 相关模块设计文档。

## 2. 基本原则
1. 文档语言以中文为主，信号名、函数名、类型名保持代码原名。
2. 先给结构和边界，再给细节，避免“代码逐行翻译式”描述。
3. 使用参数化位宽描述（如 `ISSUE_WIDTH * UOP_BITS`），避免 Magic Number。
4. 所有行为描述必须可映射到代码中的函数、状态机或信号。

## 3. 固定章节结构
每个模块文档至少包含以下章节：

1. 概述（模块职责、流水线位置、设计目标）
2. 接口定义（输入/输出信号表）
3. 微架构设计（逻辑分区、核心工作流、状态机）
4. 组合逻辑说明（`comb_xxx`）
5. 性能计数器
6. 资源占用（寄存器/存储/并发能力）
7. 已知限制与后续优化（可选但推荐）
8. 特定模块需要详细描述机制，比如重命名恢复

## 4. 接口定义要求
接口表至少包含以下字段：

| 字段 | 说明 |
| :--- | :--- |
| 信号名 | 与代码一致 |
| 位宽 | 尽量参数化 |
| 方向 | 输入/输出 |
| 来源/去向 | 上下游模块 |
| 描述 | 语义与时序要点 |

## 5. `comb_xxx` 文档要求
对每个 `comb_xxx` 必须单独建小节，且至少包含以下四项：

1. 功能描述：一句话说明核心作用。
2. 输入依赖：该 `comb` 读取或依赖的关键信号。
3. 输出更新：该 `comb` 可能更新的信号或状态。
4. 约束/优先级：冲突处理、仲裁优先级、同拍限制等。

推荐模板：

```md
### 4.x `comb_xxx`
- 功能描述：...
- 输入依赖：`sig_a`, `sig_b`, `sig_c`
- 输出更新：`out_x`, `out_y`
- 约束/优先级：...
```

## 6. 时序与异常场景要求
文档应至少覆盖以下场景：

1. 正常路径（steady state）
2. backpressure/stall 路径
3. flush/redirect 路径
4. 异常/错误路径（如 page fault、非法状态）

推荐使用“场景矩阵 + 逐拍示例”的形式，避免只有抽象描述。

## 7. 质量检查清单
提交文档前应自检：

1. 是否所有关键 `comb_xxx` 均有小节？
2. 是否每个 `comb_xxx` 都明确了“输入依赖/输出更新”？
3. 是否存在无法映射到代码的描述？
4. 是否标注了关键优先级和同拍限制？
5. 是否覆盖异常与 flush 行为？
