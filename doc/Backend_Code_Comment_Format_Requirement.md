# Back-end 代码注释格式要求

## 1. 适用范围
本规范适用于 back-end 代码注释，覆盖头文件与实现文件，重点约束 `comb_xxx` 注释。

## 2. 基本原则
1. 注释语言以中文为主。
2. 信号名、函数名、类型名保持英文原名，不翻译、不重命名。
3. 注释服务“快速理解行为与边界”，避免重复代码表面语义。
4. 注释必须与实现保持一致，改行为时同步更新注释。

## 3. 注释层级要求
建议按三层组织注释：

1. 文件级：模块职责、边界、关键数据通路。
2. 函数级：函数目的、输入依赖、输出更新、关键约束。
3. 复杂代码块级：仅对难以直读的决策逻辑补充说明。

## 4. `comb_xxx` 强制注释格式
每个 `comb_xxx` 函数定义上方必须有注释，且至少包含以下四项：

1. 功能：该 `comb` 的核心职责。
2. 输入依赖：读取/依赖的关键信号。
3. 输出更新：可能驱动或修改的信号。
4. 约束：优先级、互斥关系、同拍条数限制等。

推荐模板：

```cpp
// comb_xxx
// 功能: ...
// 输入依赖: sig_a, sig_b, sig_c
// 输出更新: out_x, out_y
// 约束: ...
void comb_xxx();
```

## 5. 注释内容要求
1. 输入依赖应列“决定行为”的信号，不要求罗列所有局部临时变量。
2. 输出更新应列“对外可见或影响后续逻辑”的信号。
3. 对 `valid/ready`、仲裁、旁路、flush 覆盖的优先关系必须写清。
4. 若函数存在“无副作用”或“仅计算中间量”，要显式注明。

## 6. 不推荐写法
1. 逐行翻译代码（如“给变量赋值”）。
2. 空泛注释（如“处理逻辑”）。
3. 与实现不一致的历史注释。
4. 在注释中使用与代码不一致的别名。

## 7. 最小检查清单
对每个新增或修改的 `comb_xxx`，提交前检查：

1. 是否有函数级注释？
2. 是否明确了输入依赖？
3. 是否明确了输出更新？
4. 是否注明关键约束/优先级？
5. 注释是否与当前代码一致？
